# 结构化系统提示与缓存策略 Spec

## 背景

MewCode 当前通过三条简短指令区分普通执行、规划和计划执行模式，并在每次模型请求中把模式指令作为一个整体交给 Provider。全局行为规则、工具约定、当前模式和运行环境尚未形成统一的结构化提示体系。

这种扁平结构存在几个限制：

- 稳定指令与动态内容缺少明确边界，无法有意识地维护可复用的缓存前缀。
- 新增规则时缺少固定的职责划分和优先级，容易产生重复、顺序漂移或 Provider 行为差异。
- 多轮工具执行期间缺少按频率强化当前模式的机制。
- 工具描述与全局规则之间没有统一的关键约定，模型可能忽略专用工具或跳过编辑前读取。
- 缓存命中数据没有进入统一的用量观测通道，无法证明缓存策略是否生效。

本里程碑将在保留现有双 Provider、流式响应、工具循环和三种任务模式行为的基础上，引入结构化提示与缓存边界。

## 目标

- 建立按优先级组合的系统提示体系，使固定模块、环境信息和可选扩展模块拥有明确顺序与职责。
- 将稳定指令及工具描述与动态环境、对话历史和当轮补充指令分离，使 Provider 能采用各自支持的缓存机制。
- 使用统一的 `<system-reminder>` 系统级补充消息，在运行中强化模式和动态约束，同时不写入会话历史。
- 在全局指令与相关工具描述中双重表达关键规则，提高模型遵守工具选择和编辑流程约定的概率。
- 统一解析并展示缓存读写指标，以自动化测试覆盖两个 Provider，并通过至少一个具备凭据的 Provider 完成实网命中验证。
- 准备典型任务场景，用于人工比较改造前后的指令遵守、工具选择和输出质量。

## 功能需求

- **F1：固定模块与顺序**
  系统必须提供七个固定模块，并严格按以下优先级排列：身份、系统约束、任务模式、动作执行、工具使用、语气风格、文本输出。模块之间以一个空行分隔；相同输入必须生成相同顺序与内容。

- **F2：动态与可选模块顺序**
  七个固定模块之后依次追加环境信息、自定义指令、已激活的 Skill、长期记忆。后三项为可选模块，无内容时整体省略，不产生空标题或多余分隔。当前里程碑只接受调用方直接提供的内容，不负责从文件、Skill 系统或记忆系统加载。

- **F3：稳定与动态通道分离**
  固定模块和工具描述必须进入稳定、可缓存的请求区域；环境信息、当前激活模式的补充指令、可选模块及对话历史通过动态消息区域发送。动态内容变化不得改变稳定内容本身或其前缀顺序。

- **F4：双 Provider 缓存映射**
  OpenAI 和 Anthropic 必须把同一份结构化提示映射到各自原生支持的系统指令、消息与缓存机制。相同稳定指令和相同工具集合应形成可复用的缓存输入；工具集合或稳定指令变化时不得错误复用不兼容缓存。

- **F5：系统级补充消息**
  每轮模型请求必须能够携带 `<system-reminder>补充内容</system-reminder>` 补充消息。该消息具有系统指令语义，不得作为用户输入提交，不得写入会话历史，也不得作为一条用户消息显示或诱导模型专门回复。

- **F6：模式指令注入频率**
  每个 Agent 运行独立计数模型请求轮次。第 1 轮以及之后每间隔 5 轮，即第 1、6、11……轮，注入完整模式指令；其他轮次注入精简提醒。每次新运行从第 1 轮重新计数，普通执行、规划和计划执行模式均遵循同一频率规则。

- **F7：环境快照**
  每个运行开始时生成一次最小环境快照，并在该运行各轮中保持一致。快照只包含当前工作目录、操作系统/平台、当前交互 Shell、当前日期和时区；不得扫描 Git、列出目录、读取任意环境变量集合或包含配置与密钥。

- **F8：关键工具规则双重强化**
  “优先使用匹配任务的专用工具”和“编辑前必须先读取目标内容”等关键规则，必须同时出现在全局工具使用模块和相关工具描述中。各工具描述只重复与自身相关的规则，避免加入互相矛盾或无关的约束。

- **F9：工具定义稳定性**
  相同工具作用域下，工具名称、描述、参数结构和顺序必须稳定。规划模式的只读工具集合与可执行模式的完整工具集合应被视为不同的缓存输入。

- **F10：缓存用量归一化**
  两个 Provider 返回的缓存读取量和缓存写入量必须进入统一用量模型，并与每轮及累计 token 用量一起传播。Provider 未返回某一指标时必须保留“不可用”语义，不得伪装为零。

- **F11：缓存指标展示**
  当缓存指标可用时，全屏 TUI 与纯文本模式必须在现有 Usage 信息中显示缓存读取量和缓存写入量；均不可用时保持当前显示形式，避免增加无意义字段。

- **F12：缓存策略验证**
  自动化测试必须覆盖两个 Provider 的请求结构、稳定前缀、工具缓存属性、补充消息位置及缓存字段解析。人工实网验证至少使用一个具备有效凭据的 Provider，通过重复等价请求观察非零缓存读取量。

- **F13：人工对比场景**
  本里程碑必须提供一组可重复执行的典型场景，至少覆盖专用工具优先、编辑前读取、规划模式只读、多轮模式提醒、输出风格和重复请求缓存命中，并记录改造前后可比较的观察项。

## 非功能需求

- **N1：确定性与缓存友好性**
  相同固定模块和工具集合必须生成字节级一致的可缓存内容。日期、路径、轮次编号等动态数据不得进入稳定区域。

- **N2：现有行为兼容**
  改造不得破坏现有三种任务模式、工具作用域、流式输出、工具循环、确认流程、取消语义和会话历史提交规则。不得新增用户必须配置的字段。

- **N3：Provider 一致性与渐进降级**
  两个 Provider 对外呈现同一套提示和用量语义。某个模型或兼容服务不支持缓存、未达到缓存门槛或未返回缓存字段时，请求仍应正常完成，并明确表现为指标不可用或未命中。

- **N4：安全与隐私**
  系统提示、环境快照、用量显示、错误信息和测试证据不得包含 API key、完整环境变量、配置密文或其他凭据。人工实网验证也必须沿用现有密钥脱敏约定。

- **N5：上下文开销受控**
  精简模式提醒必须明显短于完整版本；空模块必须完全省略。提示构建不得扫描仓库或发起额外网络请求，新增本地处理时间相对于一次模型请求应可忽略。

- **N6：可扩展性**
  新增固定模块、动态补充模块或可选模块时，应能通过明确的优先级和通道属性插入，而不需要复制整套拼装逻辑，也不应要求两个 Provider 分别维护内容顺序。

- **N7：可测试性**
  提示拼装、轮次频率、Provider 请求映射、缓存字段解析和用量累计必须能够使用确定性输入及模拟响应测试。需要真实凭据的验证不得成为默认自动化测试的前置条件。

- **N8：观测真实性**
  “未返回”“返回零”和“返回正数”必须保持不同语义。只有实际解析到非零缓存读取量时，才能认定某次请求命中缓存；不得仅凭请求结构正确就宣称缓存生效。

- **N9：错误隔离**
  缓存字段缺失不得导致请求失败；字段格式非法或 Provider 响应协议损坏时，应通过现有 Provider 错误通道报告，且不得提交不完整的会话历史或错误累计用量。

- **N10：文档一致性**
  规格、设计、任务和验收文档必须使用一致的模块名称、顺序、缓存术语和轮次定义；人工对比场景必须能够由另一位执行者按文档重复。

## 不做的事

- 不加载 `AGENTS.md`、项目级提示文件或其他仓库指令文件；本章只预留自定义指令插槽。
- 不实现 Skill 的发现、选择或激活流程；只允许调用方直接提供已激活 Skill 的文本。
- 不实现长期记忆的生成、检索、排序或持久化；只预留长期记忆模块。
- 不接入真实 MCP Server，也不改变现有内置工具注册和执行体系。
- 不实现自动化行为评分、LLM-as-judge、基准测试平台或统计显著性分析；本章只提供人工对比场景。
- 不增加本地响应缓存、跨 Provider 缓存或自行托管的缓存服务；这里只使用 Provider 原生的 Prompt Caching。
- 不保证所有请求必然命中缓存；模型能力、最小 token 门槛、缓存过期和 Provider 调度均可能导致未命中。
- 不新增缓存诊断命令、独立面板或 TUI 布局重构；缓存指标沿用现有 Usage 展示。
- 不新增提示模块、缓存周期或环境字段的用户配置入口；本里程碑使用已批准的固定顺序和频率。
- 不扫描 Git 状态、仓库文件、目录内容或任意环境变量来扩充环境快照。
- 不把“编辑前读取”等提示约定升级为新的运行时拦截器；本章只通过全局提示与工具描述双重强化，现有工具自身的安全检查保持不变。
- 不扩展到 OpenAI 和 Anthropic 之外的其他 Provider，也不更改模型选择、价格配置或 API key 管理方式。
- 不新增对话历史持久化，也不把系统补充消息保存到现有会话历史中。

## 验收标准

- **AC1（F1）**：给定全部固定模块，生成结果严格按“身份 → 系统约束 → 任务模式 → 动作执行 → 工具使用 → 语气风格 → 文本输出”排列，模块之间恰好保留一个空行。

- **AC2（F2）**：给定环境信息及三个可选模块，生成结果在固定模块之后严格按“环境信息 → 自定义指令 → 已激活 Skill → 长期记忆”排列；任一可选模块为空时，其标题、内容和分隔均不出现。

- **AC3（F3、N1）**：分别改变工作目录、日期、当前模式提醒、对话历史和轮次编号后，可缓存内容保持字节级一致，变化只出现在动态请求区域。

- **AC4（F2）**：调用方直接提供自定义指令、Skill 或长期记忆文本时，每段内容只出现一次；未提供加载来源时系统不主动读取任何文件或外部数据。

- **AC5（F4）**：连续构造两次具有相同固定模块和工具集合、但环境或用户输入不同的 OpenAI 请求，可观察到相同的稳定前缀，动态内容位于其后，且补充指令未被编码为用户输入。

- **AC6（F4）**：连续构造两次具有相同固定模块和工具集合、但环境或用户输入不同的 Anthropic 请求，可观察到稳定系统内容和工具定义带有正确缓存边界，动态内容位于缓存边界之后，且补充指令未被编码为用户输入。

- **AC7（F5）**：每次模型请求都能观察到格式完整的 `<system-reminder>补充内容</system-reminder>`；一次运行结束后，会话历史和用户界面消息列表中均不存在该补充消息。

- **AC8（F6）**：模拟一次 10 轮运行，第 1、6 轮收到完整模式指令，第 2–5、7–10 轮收到精简提醒；开始下一次运行时重新从完整版本开始。

- **AC9（F7、N4）**：环境快照只包含工作目录、平台、交互 Shell、日期和时区；检查生成内容后看不到 Git 状态、目录列表、API key、配置密文或其他环境变量。

- **AC10（F8）**：检查全局工具使用模块和相关工具描述，可同时找到“优先使用专用工具”与“编辑前先读取”的一致约定；不相关工具不携带无关规则。

- **AC11（F9）**：相同工具作用域重复生成定义时，名称、描述、参数结构和顺序完全一致；规划模式只读集合与执行模式完整集合产生不同且各自稳定的缓存输入。

- **AC12（F10）**：输入包含 OpenAI 缓存读取、缓存写入和普通 token 字段的模拟响应后，统一用量结果保留对应数值；字段缺失与字段值为零得到不同结果。

- **AC13（F10）**：输入包含 Anthropic 缓存创建、缓存读取和普通 token 字段的模拟响应后，统一用量结果保留对应数值；字段缺失与字段值为零得到不同结果。

- **AC14（F10、N8）**：模拟多轮混合用量后，每轮和累计输入、输出、总量、缓存读取及缓存写入均计算正确；任一未知维度不会被当作零累计。

- **AC15（F11）**：缓存指标存在时，全屏 TUI 与纯文本模式的 Usage 行均显示缓存读取和写入量；指标均不存在时，两种界面保持原有 Usage 显示形式。

- **AC16（F12、N7）**：不访问真实网络即可运行双 Provider 自动化测试，验证稳定前缀、工具缓存属性、系统补充消息位置、轮次频率和缓存字段解析。

- **AC17（F12、N8）**：使用至少一个具备有效凭据且支持 Prompt Caching 的 Provider，连续发送满足缓存门槛的等价稳定前缀请求；后续请求的统一 Usage 中观察到缓存读取量大于零，方可判定实网缓存验证通过。

- **AC18（F13）**：人工对比清单至少包含专用工具优先、编辑前读取、规划模式只读、多轮模式提醒、输出风格和重复请求缓存命中六类场景；每个场景都有固定输入、观察项以及改造前后结果记录。

- **AC19（N2）**：现有普通执行、规划、计划执行、流式输出、工具反馈、确认、取消、错误恢复和历史提交测试全部通过，且不要求新增用户配置。

- **AC20（N3）**：模拟 Provider 不支持缓存或完全不返回缓存字段时，请求正常完成，界面不显示虚假的缓存指标。

- **AC21（N4、N9）**：模拟非法缓存字段或损坏的 Provider 响应时，错误经现有 Provider 错误通道返回，不泄露凭据、不提交当前不完整历史，也不污染累计用量。

- **AC22（N5）**：构建提示和环境快照期间未发生仓库扫描或额外网络请求；空模块被省略，精简模式提醒的内容长度小于完整版本。

- **AC23（N6）**：加入一个仅用于测试的可选模块后，可以通过优先级插入正确位置，无需分别修改 OpenAI 与 Anthropic 的内容排序逻辑。

- **AC24（N10）**：`spec.md`、后续设计、任务与验收文档对模块名称、模块顺序、“模型请求轮次”、缓存读取和缓存写入的定义保持一致。
