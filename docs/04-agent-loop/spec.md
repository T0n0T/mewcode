# MewCode Agent Loop Spec

## 背景

MewCode 已具备多轮对话、双 Provider 流式响应、六个工作区工具、工具结果回灌、安全确认、用户取消和全屏终端界面。

当前运行时仍是固定的两段流程：模型首次响应最多执行一个工具，随后只能生成最终文本；若任务需要连续读取、搜索、修改和验证，用户必须反复催促模型继续。一次返回多个工具调用时也会全部拒绝。与此同时，文本事件与工具交互仍通过不同通道传递，执行链路以同步接口为主，限制了 Agent 编排与界面的独立演进。

本里程碑将固定两段流程升级为可自主推进的 Agent Loop，使 MewCode 能在明确安全边界内持续观察工具结果、调整下一步行动，直到任务完成或触发停止条件。

## 目标

- 建立 ReAct 风格的自主循环，反复完成“请求模型、收集响应、执行工具、回写结果”，直到模型给出不再调用工具的最终回答。
- 覆盖模型自然完成、10 轮迭代上限、用户取消、连续 3 轮请求未知工具和模型流错误等停止条件，并向界面提供明确停止原因。
- 将 Provider、Agent 编排、工具执行和界面消费升级为端到端异步链路。
- 建立统一异步事件流，覆盖文本、工具调用、工具结果、Token 用量、进度和终止状态，使 Agent 行为不依赖具体界面。
- 在实时推送文本的同时收集完整模型响应，兼顾低延迟显示与后续工具判断、历史提交。
- 支持一次响应中的多个工具调用：显式标记可并发或必须串行，保持原始顺序，以串行调用作为批次屏障。
- 提供两段式 Plan Mode：`/plan <任务>` 仅使用读类工具形成并保存计划，`/do` 使用全部工具执行最近计划。
- 保留既有工作区边界、密钥保护和副作用工具确认机制；本阶段不新增权限系统、上下文压缩或新的用户交互式确认能力。

## 功能需求

- **F1：统一 Agent 运行模式。** 普通用户请求进入全工具执行模式；`/plan` 进入只读规划模式；`/do` 使用全工具执行已保存计划。三种入口复用相同的循环、事件和停止语义。
- **F2：自主 ReAct 循环。** 每轮请求模型并收集完整响应；若模型提出工具调用，则执行对应批次、回写全部结果并开始下一轮；若没有工具调用，则以该响应完成本次运行。
- **F3：双路流式收集。** 模型文本片段到达后立即进入事件流，同时累积完整文本、工具调用和协议状态，供响应判断、工具回写和历史提交使用。
- **F4：完整响应判定。** 只有模型流正常结束后，系统才能判断并执行其中的工具调用。流未完整结束时，不执行已收集但尚未确认完整的调用，也不把残缺助手响应提交到会话历史。
- **F5：统一异步事件流。** 每次运行通过同一条异步事件流输出文本增量、工具调用、工具结果、确认状态、Token 用量、进度变化和最终停止状态；界面无需读取 Agent 内部状态或依赖工具执行器的独立通知通道。
- **F6：进度事件。** 运行期间持续报告当前模式、当前轮次、10 轮上限及当前阶段，包括等待模型、接收响应、执行工具、等待确认、回写结果和结束。
- **F7：Token 用量。** 每次模型响应结束后报告该轮可获得的输入、输出和总 Token 用量，并同时报告本次运行的累计值；Provider 未提供的维度必须明确标记为不可用，不得推测或伪造。
- **F8：模型自然完成。** 模型返回完整响应且不再请求工具时，系统提交完整助手消息、发出完成事件并结束，不再追加模型调用。
- **F9：迭代上限。** 普通请求、`/plan` 和 `/do` 默认最多运行 10 轮。第 10 轮仍产生工具调用时，可以完成该轮已开始的工具批次及结果记录，但不得发起第 11 次模型请求，也不得额外追加一次总结请求；系统以“达到迭代上限”结束。
- **F10：用户取消。** 用户取消后应停止活动模型流，取消尚未开始或可取消的工具任务，不再启动新工具或模型请求，并发出“用户取消”停止状态。已经开始的副作用不承诺自动回滚，残缺助手文本不进入历史。
- **F11：模型流错误。** 任意模型流连接、解析或协议完整性错误都会立即停止本次运行；已显示文本可以保留供用户查看，但残缺响应和未完成工具调用不进入历史或执行，系统不自动重试。
- **F12：连续未知工具。** 未知工具调用作为结构化错误回写，让模型有机会修正。连续 3 轮的工具批次均只包含未知工具时停止；出现至少一个已知工具或模型给出最终回答时，连续计数清零。同一轮包含多个未知工具仍只计为一次。
- **F13：多工具调用。** 一次完整模型响应可以包含多个工具调用。每个调用独立完成名称、参数和调用标识校验；无效或未知调用产生对应结构化错误，不阻止同批次中的其他有效调用执行。
- **F14：显式执行策略。** 每个工具必须声明为可并发或必须串行；未声明时按必须串行处理。读取、文件查找和内容搜索允许并发，写入、修改和命令执行必须串行。
- **F15：保序批次调度。** 系统保持模型给出的调用顺序，将相邻可并发调用组成并发批次，将每个串行调用作为顺序屏障。后续批次必须等待前一批次全部结束后才能开始。
- **F16：工具结果回写。** 成功、失败、拒绝、超时、无效参数和未知工具均转换为结构化结果。并发调用可以按实际完成时间产生进度事件，但回写模型的结果必须按原始调用顺序排列；普通工具失败本身不终止循环。
- **F17：保留既有确认。** 写文件、改文件和执行命令仍逐次等待用户确认。等待与确认过程必须异步，不阻塞其他界面活动；拒绝后不产生对应副作用，并把拒绝结果回写给模型，循环可以继续调整。
- **F18：只读规划。** `/plan <任务>` 启动完整 Agent Loop，但只向模型开放读取文件、查找文件和搜索内容三类工具。命令、写入和修改工具不开放，即使命令内容看似只读也不例外。自然完成的最终回答保存为当前会话的最新计划。
- **F19：计划替换。** 只有成功完成的 `/plan` 才替换当前计划；规划被取消、流出错或因安全停止条件结束时，已有计划保持不变。
- **F20：计划执行。** `/do` 不接受新的任务正文，而是以最近一次成功且尚未完成的计划为执行目标，并恢复全部工具。不存在可执行计划或计划已经成功执行时，应直接返回明确提示，不调用模型或工具。
- **F21：计划生命周期。** `/do` 自然完成后将计划标记为已完成，重复 `/do` 不得再次执行。取消、模型流错误、连续未知工具或达到迭代上限时，计划保持可重试。普通请求不会覆盖、消费或完成已保存计划。
- **F22：会话连续性。** 只有协议完整的模型响应和已完成的工具结果才能提交到会话历史；取消、流错误和未完成批次不得留下无法被 Provider 正确重放的残缺消息。

## 非功能需求

- **N1：端到端异步。** Provider 流、Agent 编排、工具执行、确认等待和界面消费均使用异步契约；网络、文件、子进程或确认等待不得阻塞事件循环和界面响应。
- **N2：低延迟与背压。** 文本片段到达后应在下一次可见刷新周期内进入事件流，不得等待完整回复。事件传递必须具备背压或有界缓冲，慢速消费者不得造成无界内存增长。
- **N3：事件顺序稳定。** 每次运行具有明确的开始和唯一终止事件；工具结果不得先于对应调用事件，终止后不得再产生迟到事件。并发任务的身份必须保持稳定，不能因完成顺序不同而错配结果。
- **N4：界面隔离。** Agent、Provider 和工具层不得依赖具体终端界面。全屏界面、纯文本界面或未来其他客户端只能通过统一事件流和必要的异步控制入口参与运行。
- **N5：取消可靠性。** 取消操作必须传播到活动模型流、待执行批次、并发工具任务和确认等待；系统应及时释放网络流、子进程及后台任务，不留下影响后续运行的孤立任务。
- **N6：并发安全。** 同一批可并发工具可以重叠执行，但任何标记为串行的工具都不得与同一响应中的其他工具重叠。并发优化不得改变批次屏障、确认顺序或模型收到的结果顺序。
- **N7：协议隔离。** OpenAI 与 Anthropic 的原生流事件、工具格式、Token 用量和历史回放细节只能由对应 Provider 处理；Agent Loop 只依赖统一语义。
- **N8：双 Provider 一致性。** 两个 Provider 都必须支持多轮工具循环、多工具调用、异步取消、Token 用量、计划模式和完整历史重放；协议能力缺失时应明确降级，不得静默伪造数据。
- **N9：历史一致性。** 助手响应与对应工具结果应以协议可重放的完整单元提交。并发、取消或异常不得造成孤立调用、重复结果或部分提交。
- **N10：安全与密钥保护。** 工作区逃逸防护、工具超时、结果截断和副作用确认继续生效。事件、进度、错误、工具参数及结果不得泄露 API key；面向界面的工具结果只包含经过脱敏和限制的安全展示数据。
- **N11：错误隔离与恢复。** 单个工具失败不得使整个会话崩溃；模型流错误只终止当前运行。终止后界面和会话必须能够接受新的普通请求、`/plan` 或可重试的 `/do`。
- **N12：可观测性。** 每个事件都应具备足以关联运行、轮次、工具调用和批次的稳定信息。任何停止都必须有机器可判定的原因和用户可理解的安全说明。
- **N13：Token 统计一致性。** 单轮用量与累计用量必须来自 Provider 返回的真实数据，并满足累计值等于各轮已知值之和；缺失字段不能被当作零参与误导性统计。
- **N14：可测试性。** 循环、并发顺序、取消、确认、Token 用量、停止条件和计划生命周期均可通过替代 Provider、替代工具及可控时间源自动验证，无需真实网络、危险命令或不稳定等待。
- **N15：行为兼容。** 既有六个工具的参数、结果、安全边界和确认规则保持不变；普通聊天、配置查找顺序、双 Provider、全屏与纯文本终端、`exit`、`quit` 和中断行为继续可用。
- **N16：会话级状态。** 保存的计划、执行状态、Token 累计值和 Agent 进度只存在于当前进程与当前会话，程序重启后不恢复。
- **N17：配置稳定。** 本里程碑不新增用户配置字段，不改变现有 Provider 配置格式、文件位置或优先级。10 轮上限允许由运行时调用方覆盖，仅用于嵌入和测试。

## 不做的事

- 不实现项目级、用户级或按工具配置的权限系统、白名单、信任级别或自动批准策略。
- 不新增“询问用户”“请求澄清”一类工具或通用人机交互协议；只保留现有副作用工具确认。
- 不移除、绕过或自动批准写文件、改文件和执行命令的既有确认。
- 不实现上下文压缩、自动摘要、上下文窗口裁剪、历史淘汰或 Token 预算管理。
- 不根据 Token 用量估算费用、限制消费或新增成本仪表盘；本阶段只报告 Provider 返回的用量。
- 不持久化会话、计划、事件、进度、Token 用量或中断点，也不支持程序重启后恢复执行。
- 不实现 Agent 运行快照、检查点、事件重放或崩溃恢复。
- 不实现多 Agent、子 Agent、任务委派、后台 Agent 或多个用户请求同时运行。
- 不支持生成期间提交、排队或 steering 新消息；同一会话同时最多运行一个普通请求、`/plan` 或 `/do`。
- 不引入通用工作流引擎、任务图、计划 DSL、依赖推理或持久化任务状态机。
- 不提供计划编辑器、计划逐项勾选或额外的计划审批步骤；`/do` 直接使用最近一次成功计划，具体副作用仍按现有规则确认。
- 不自动推断命令是否“看起来只读”；命令在 Plan Mode 中始终不可用，在执行模式中始终按串行副作用工具处理。
- 不重新排列跨越串行屏障的工具调用，也不并发执行写入、修改或命令工具。
- 不自动重试模型流、失败工具或被拒绝工具；模型可以依据结构化结果在剩余轮次内自行调整。
- 不为已开始的文件修改或命令提供事务、撤销或自动回滚。
- 不新增工具、插件系统、远程工具协议、网络浏览或外部搜索能力。
- 不改变现有工具参数、工作区边界、超时、输出截断或 shell 语义。
- 不把工具内部 stdout、文件读取过程或搜索过程改造成增量流；本阶段只在工具完成后产生结果事件。
- 不新增模型或 Provider 切换、会话内配置编辑、Agent 参数面板或迭代上限配置项。
- 不重做现有终端视觉设计；只增加消费统一事件流、显示新进度与停止原因，以及识别 `/plan`、`/do` 所需的界面行为。
- 不改变配置文件位置、查找优先级或现有字段。
- 不提供操作系统级沙箱、容器隔离、网络隔离、权限降级或跨进程资源限制。

## 验收标准

- **AC1（F1、F2）：** 普通请求触发至少三轮不同工具调用时，MewCode 无需用户发送“继续”即可依次执行、回写结果并根据新观察调整后续调用，最终输出不含工具调用的回答。
- **AC2（F3、F4）：** 模型缓慢发送文本和工具参数时，文本在流结束前持续可见；流正常结束后系统能得到与所有片段一致的完整文本和调用。若流提前断开，已收集调用不会执行，残缺响应不进入历史。
- **AC3（F5）：** 一次包含模型响应和工具执行的运行只通过一条异步事件流观察到文本、调用、结果、确认、用量、进度和终止状态；更换全屏、纯文本或测试消费者不改变 Agent 行为。
- **AC4（F6）：** 每轮运行都能观察到当前模式、`当前轮次/10` 以及等待模型、接收响应、执行工具、等待确认、回写结果和结束等实际阶段，阶段顺序与真实行为一致。
- **AC5（F7、N13）：** 模拟三次带用量数据的模型响应时，每轮事件与 Provider 原始数据一致，累计值等于三轮已知值之和；缺失的用量维度显示为不可用而不是零或估算值。
- **AC6（F8）：** 模型首轮或任意后续轮返回无工具调用的完整回答时，该回答只提交一次，运行发出自然完成状态，且不再请求模型。
- **AC7（F9）：** 模型连续 10 轮都请求工具时，第 10 轮批次完成后以迭代上限停止；可以确认没有第 11 次模型请求，也没有额外总结请求。
- **AC8（F10）：** 在等待首个模型片段和接收部分文本两种情况下取消，活动流都会停止，不再出现新片段；已有文本可见但不进入历史，运行以用户取消结束。
- **AC9（F10、N5）：** 在并发读工具、串行副作用工具和确认等待期间分别取消时，尚未开始的工作不会启动，可取消任务被结束，不再请求模型；已开始副作用不会被错误描述为已回滚，终止后没有遗留任务影响下一次运行。
- **AC10（F11）：** 模型流发生连接错误、事件解析错误、重复完成信号或缺少完成信号时，本次运行立即以流错误结束，不执行未确认完整的调用、不自动重试，并可正常开始下一次请求。
- **AC11（F12）：** 模型连续三轮只请求不存在的工具时，前两轮收到结构化错误并获得纠正机会，第三轮后停止；在第二轮插入一个已知工具调用后，未知工具连续计数被重置。
- **AC12（F13）：** 一次响应同时包含多个有效调用、一个参数错误调用和一个未知工具时，所有有效调用均执行，两个无效调用分别得到结构化错误，调用标识和结果不会互相错配。
- **AC13（F14）：** 读取、文件查找和内容搜索被识别为可并发；写入、修改和命令被识别为串行；测试中未声明策略的工具按串行处理。
- **AC14（F15、N6）：** 调用顺序为“读 A、读 B、写 C、读 D”时，可观察到 A/B 重叠执行，C 只在 A/B 全部结束后开始，D 只在 C 结束后开始；任何串行工具都不与其他调用重叠。
- **AC15（F16）：** 并发工具可以按不同顺序完成并实时报告状态，但模型收到的结果仍按原始调用顺序排列；成功、错误、拒绝、超时、无效参数和未知工具使用一致的结构化表达。
- **AC16（F17）：** 同一响应包含多个副作用工具时，每个工具按顺序单独确认。批准后才执行，拒绝后无对应副作用；批准或拒绝结果均回写模型，剩余轮次仍可继续。
- **AC17（F18）：** `/plan <任务>` 可以连续调用读取、查找和搜索工具形成计划；模型看不到写入、修改或命令工具，最终计划成功保存。尝试按名称调用未开放工具时按未知工具规则处理。
- **AC18（F19）：** 已存在计划 A 时，一次成功的 `/plan` 会用计划 B 替换它；一次被取消、流出错、达到上限或连续请求未知工具的规划不会覆盖当前有效计划。
- **AC19（F20）：** 存在未完成计划时，`/do` 使用该计划和全部工具启动执行；没有计划或计划已完成时，直接显示明确提示，模型和工具调用次数均为零。
- **AC20（F21）：** `/do` 自然完成后再次输入 `/do` 不会重复执行；取消、流错误、未知工具停止或迭代上限后可以重试同一计划；期间执行普通请求不会改变计划状态。
- **AC21（F22、N7、N8、N9）：** OpenAI 与 Anthropic 都能重放包含多轮助手调用及多个工具结果的完整历史；取消、流错误和未完成批次不会产生孤立调用、重复结果或 Provider 拒绝的消息序列。
- **AC22（N1、N2）：** 模型流、并发工具和确认等待运行期间，界面仍能刷新、编辑草稿和处理取消；大量快速文本片段在慢速消费者下不丢失、不乱序，也不会形成无界待处理事件。
- **AC23（N3、N4、N12）：** 每次运行恰好产生一个终止事件，终止后迟到的模型或工具事件被忽略；所有运行、轮次、批次和调用均可稳定关联，并能机器判定自然完成、迭代上限、用户取消、未知工具或流错误。
- **AC24（N10）：** 工具参数、结果、错误、进度和界面事件均不包含测试 API key；读取与搜索的完整大结果不会未经限制地进入展示事件，工作区越界、超时、截断和副作用确认继续生效。
- **AC25（N11）：** 工具失败后模型可以在剩余轮次调整方案；模型流错误只终止当前运行。两种情况下界面均保持可用，并能成功完成下一次普通请求或计划操作。
- **AC26（N14）：** 所有循环、并发、停止、Token、确认和计划生命周期场景均可使用可控替代组件自动验证，不需要真实 Provider、真实危险命令或基于实际时间的等待。
- **AC27（N15）：** 既有六个工具、普通无工具对话、配置优先级、OpenAI、Anthropic、全屏模式、纯文本模式、`exit`、`quit` 和中断行为全部通过回归验证。
- **AC28（N16、N17）：** 退出并重新启动后不恢复计划、Agent 进度或用量累计；现有配置文件无需新增字段即可启动，查找位置与优先级保持不变。
